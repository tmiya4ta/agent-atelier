Broker-centric Agent Redesign

SCRS demo の各 agent から横断的な MDM lookup tool を取り上げ、 broker の存在価値を可視化する。 各 agent は自分の専門業務にしか tool を持たず、 知らない情報が必要になったら [NEED_INFO] マーカーで明示的に問い返す。 人間が往復していたエージェント間連携を broker が肩代わりする ── という流れを demo で見せる。

1. なぜこの設計に変えるのか

既存設計では各 agent が shared 9 個 の MDM lookup tool (resolve_supplier / list_suppliers / resolve_part / list_parts / list_part_alternatives / find_suppliers_for_part / get_supplier_relations / list_regions / list_incidents) を全員ぶら下げている。 これだと agent は自分一人で大半の参照が完結してしまい、 broker の出番が無い。 demo として broker の価値が伝わらない。

ねらい: 各 agent を「専門業務の executor」に純化する。 lookup は持たない、 横断判断もしない。 足りない情報があれば [NEED_INFO] で人間 (= 上位の orchestrator) に問い返す。 broker がいれば、 これを自動的に他 agent から集めてくれるので人間は最初の指示と最後の結果だけ見ればよい。 broker がいなければ、 人間が agent 間を行き来する手間が visible になる。

3. 人間への問い直しプロトコル

各 agent の system prompt に追記し、 LLM の自然な発話で問い返しを生成させる。 tool として実装する案 (ask_user(question)) も検討したが、 demo の見た目が自然な (A) を採用。

system_prompt 追記文 (例)

You only have tools for your own domain ({{agent_domain}}).
When you lack cross-domain information (supplier IDs, part IDs,
inventory levels, contracts, decisions from other domains, etc.),
DO NOT guess. Output a clearly-marked block:

[NEED_INFO]
- <question 1>
- <question 2>
[/NEED_INFO]

The orchestrator (broker or human) will resolve these and reply.
Only after receiving the answers should you proceed with your tools.

marker の形式

  • 開始: [NEED_INFO]
  • 終了: [/NEED_INFO]
  • 本体: 1 行 1 質問のリスト (- ...)

broker 側はこれを正規表現 (/\[NEED_INFO\]([\s\S]*?)\[\/NEED_INFO\]/) で抽出して、 該当する別 agent に投げ直す。 人間運用時は単に画面上の質問を読んで答えるだけ。

4. Broker の役割 (今回の射程外、 後続タスク)

  • 人間からの初回 task を受け取る
  • 適切な専門 agent (incident / legal / etc.) を選び、 task を投げる
  • 応答に [NEED_INFO] があれば、 各 question を別 agent に振って回答を集約
  • 集めた回答を最初の agent に折り返して再投入
  • [NEED_INFO] が消えるまでループ (上限 N hop で打ち切り)
  • 最終結果を人間に summary 1 通で返す
broker 自体はこの doc の対象外。 agent 側を上記プロトコルに合わせて再設計するのが今回のスコープ。

5. 想定フロー (broker なし vs broker あり)

5.1 broker なし (人間が往復)

USER → incident-agent: "関西部品の P-2024-KAN-001 で不良 15%、 INC 登録して" ↓ incident-agent → USER: [NEED_INFO] - supplier ID for "関西部品" ? - normalized part ID for "P-2024-KAN-001" ? [/NEED_INFO] ↓ USER → ??? どこに聞けば... (人間が手動で MDM-like agent を巡回) USER → org-agent ? incident-agent ? lookup できる相手を探す ↓ USER → incident-agent: supplier_id=G-KANSAI-PARTS-001 / part_id=... ↓ incident-agent: register_incident → INC-2026-0527 created.

5.2 broker あり (自動連携)

USER → broker: "関西部品の P-2024-KAN-001 で不良 15%、 INC 登録して" ↓ broker → incident-agent incident-agent → broker: [NEED_INFO] supplier ID, part ID ↓ broker → mdm-agent (or any peer that can answer) mdm-agent → broker: G-KANSAI-PARTS-001 / part_id=P-... ↓ broker → incident-agent: (with resolved IDs) incident-agent: register_incident → INC-2026-0527 ↓ broker → USER: "登録完了 INC-2026-0527 です。"

→ 人間は最初と最後だけ。 broker が居る・居ないで demo の体験差がはっきり見える。

6. 実装ステップ

  1. 各 agent.xml の tools 配列から shared 9 個を削除 (5 ファイル)
  2. procurement-agent から inventory 3 tool (check_inventory / reserve_inventory / cancel_inventory_reservation) を削除
  3. 各 agent の system_prompt に [NEED_INFO] プロトコル文を追記
  4. 各 agent の pom.xml で version bump (1.2.x → 1.3.0 など)
  5. 5 agent を再 deploy (Anypoint Exchange upload + CH2 redeploy)
  6. 動作確認: 各 agent に「関西部品の P-2024-KAN-001 で不良 15%」を投げ、 [NEED_INFO] が返ることを目視
  7. scenarios/scrs-a.json に「broker なし版 (面倒)」と「broker あり版 (スマート)」 2 シナリオを書き分け

7. 確定済みの判断と検討メモ

論点決定
shared 9 個の扱い 全削除 (broker の出番を最大化)
inventory 3 tool の集約先 logistics-agent に集約 (procurement からは削除)
NEED_INFO marker の言語 [NEED_INFO] / [/NEED_INFO] 英語固定 (broker の正規表現 parse 安定性のため)
ask_user tool 化 vs prompt 誘導 prompt 誘導 (A 案) を採用
既存固有 tool (register_incident 等) 触らない
